- **M5 PR 1 (Momentum12_1Signal: JT1993 12-1 total-return momentum)** per ADR 0002 dec 20. First M5 PR; the last production `NotImplementedError` stub. Plan + Plan-reviewer + post-impl reviewer per project rule 2A. Data-independent (synthetic-fixture tested); unblocked by the S&P 500 bundle pull.
  - **`Momentum12_1Signal.compute(universe, dt, pit_view) -> dict[AssetId, float]`** real implementation. Score = 12-month total return through one month before `dt` (the JT1993 most-recent-month skip drops the short-term reversal). Window edges are calendar-resolved via `dateutil.relativedelta` (`dt - 12 months` / `dt - 1 month`) to the last available SEP bar on or before each target, NOT a fixed trading-day index count.
  - **PIT + total-return discipline**: the score is reconstructed FORWARD from raw `closeunadj` + explicit PIT ACTIONS dividends + PIT ACTIONS splits via `reconstruct_total_return`, NEVER the back-adjusted `close` (whose cumulative factor bakes in every future split/dividend up to the data-pull date, a lookahead leak per `docs/methodology/total_return_reconstruction.md:175`). Only actions whose ex-date falls inside the lookback window (all strictly before the rebalance) are applied.
  - **Split handling (beyond the Plan-reviewer's design)**: `closeunadj` is the as-traded price, so it drops by the split ratio on a split ex-date; left raw the total-return multiplier would read a spurious ~50% loss. The signal divides both the price and the dividend amount by a cumulative in-window split factor (the product of `SplitAction.ratio` for splits whose ex-date is strictly after the bar) before reconstructing, which cancels the drop. AAPL/TSLA/NVDA/GOOGL all split inside the 2005-2024 study window, so this is load-bearing for the headline study. The dividend co-adjustment reads the split factor at each ex-date via a backward `join_asof` (post-impl reviewer High 1) so a non-bar ex-date inherits the correct factor.
  - **AssetId-to-ticker resolution (Plan-reviewer Critical)**: the Signal protocol passes AssetIds (permaticker ints) but SEP is ticker-keyed and `compute` receives no resolver. The signal resolves AssetId to ticker from a `pit_view("tickers")` view using the resolver's asof interval logic (`firstpricedate <= dt <= lastpricedate`, null last = active), then joins SEP on ticker. PitView is specified to serve three tables (`sep`, `actions`, `tickers`), each sliced to `available_dt < dt` by the engine. No protocol change.
  - **Omit-not-NaN + determinism**: assets with no bar at either window edge are omitted from the result (never emitted with a NaN score) per the Signal protocol; the result is `dict(sorted(...))`.

### Fixed
- **SSGA XLSX real-structure parsing**: the first SSGA download revealed the actual export shapes differ from the initial loader assumptions. `spdr-product-data-us-en.xlsx` has a disclaimer paragraph in row 0, the group-header row (`As of**`, `Ticker`, ...) in row 1, the period sub-labels in row 2, and data from row 3; tickers carry a registered-trademark suffix (`SPY` reads as `SPY` + U+00AE, surfacing as `SPYM`/`SPY?`), which must be cleaned to avoid matching `SPYM`/`SPYG`; returns are percentage strings (`"31.01%"`) with `"-"` for not-applicable. `spdr-etf-historical-distributions.xlsx` stores ex-dates as `MM/DD/YYYY` text and dividends as whitespace-padded number-as-text. The loader now scans for the header row (handles the disclaimer offset), cleans tickers via leading-alnum regex, parses percent/number strings, and parses `MM/DD/YYYY`. Verified against Sam's real pull: 134 SPY dividends (1993-2026), SPY 10y NAV TR 15.1% as of 2026-04-30, correctly distinguished from the `SPYM`/`SPYG` rows. Synthetic test fixtures rewritten to reproduce all four quirks. New `SSGASpyReference.as_of_date` property exposes SSGA's trailing-return as-of date (needed for the window-alignment work in ADR 0006).
- **Adapter date dtype**: `SharadarDataSource.read_sep_prices` and `read_actions_dividends` now cast their date columns to `pl.Date` explicitly. The `nasdaq-data-link` SDK returns pandas `datetime64[ns]`; `pl.from_pandas` promotes that to `pl.Datetime`; `write_parquet` preserves it. Without the cast, downstream `(asset_id, dt)` price-index keys in the BarLoop become `(int, datetime)` while lookups use `dt.date()`, so every lookup silently returns `None`. The constant-weight demo then ran 3774 trading days, never rebalanced, stayed 100% in cash, and the reference function reproduced the same wrong number, so `engine_vs_reference: PASS` printed on `$0 vs $0`. This was a silent-correctness failure the synthetic-fixture tests did not catch because they wrote `pl.Date` columns directly. New regression test `test_read_sep_prices_returns_pl_date_column_even_from_datetime_source` writes a `pl.Datetime`-typed parquet and asserts the adapter returns `pl.Date`. Surfaced when Sam ran the demo against his first real Sharadar pull.
- **Manifest helper**: `sharadar_pull --refresh-hashes` now hashes `.parquet`, `.xlsx`, and `.csv` files (previously parquet-only). Required for the SSGA bundle which contains XLSX files. `row_count` is computed for parquets only; XLSX/CSV use `row_count=None` per the manifest schema. Updated `test_refresh_hashes_no_parquets_raises` to match the new error message.
