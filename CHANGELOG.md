### Fixed
- **ADR 0013 (PSR/DSR/MinTRL public API and Bailey-LdP 2014 numerical pin correction)** prep PR before M4 PR 1. Plan-reviewer surfaced three Critical findings on the M4 PR 1 plan that traced back to errors in the methodology research note + propagated into ADR 0002, ADR 0003, ROADMAP, and dataset_versioning.md.
  - **Wrong numerical pin**: `docs/research/sources/methodology-backtest-overfitting.md` worked example used incorrect `Phi_inv` quantile values (`1.869` and `1.624`; verified against `scipy.stats.norm.ppf` v1.17.1 the correct values are `1.834` and `2.249`). The resulting `SR_0 = 1.092` and `DSR = 0.971` are both wrong.
  - **Wrong sigma_sq formula**: methodology doc line 163 wrote DSR's `sigma_sq` using `SR_0` (Score / Rao form) while the PSR formula at lines 197-200 used `SR_hat` (canonical Wald form). A single coherent convention cannot make both right; Bailey-LdP 2014 publishes the Wald form, so the doc line 163 was a typo.
  - **Corrected pin**: `DSR = 0.766 within 1e-3` (Convention A canonical Bailey-LdP 2014 Wald form). Verified against `scipy.stats.norm` v1.17.1: `Phi_inv(1 - 1/30) = 1.833915`; `Phi_inv(1 - 1/(30*e)) = 2.248799`; `SR_0 = 1.311328`; `sigma_sq = 4.0` (using `SR_hat=1.5`); `z = 0.7246`; `DSR = 0.765653`.
  - **ADR 0013 locks**: corrected pin, the canonical Wald `sigma_sq = 1 - g3 * SR_hat + (g4 - 1)/4 * SR_hat^2` form, MinTRL return type AMENDED to `float` (Bailey-LdP 2012 publishes a real-valued lower bound; the ADR 0003 stub's `-> int` was a misreading), DSR `n_effective == 1` degenerates to `PSR(sr_star=0.0)` per methodology doc line 214 (does NOT raise), domain-violation behavior is `raise ValueError` consistent with codebase discipline, `_EULER_MASCHERONI` hardcoded at 16-digit precision, and the no-scipy first-principles rationale (the M4 PR 1 plan's "determinism doc says scipy is known-broken" citation was fabricated; grep verified zero scipy mentions in `docs/methodology/determinism.md`).
  - **Files corrected in place** (all with explicit ADR 0013 cross-references): `docs/decisions/0002-roadmap-review.md` lines 114 + 370; `docs/decisions/0003-architecture.md` lines 513 + 789; `docs/ROADMAP.md` line 104; `docs/methodology/dataset_versioning.md` line 139; `src/pit_backtest/analytics/sharpe.py` docstring line 33. The methodology doc's worked example now shows the corrected math step-by-step plus a pre-correction footnote so future readers see why 0.971 propagated.
  - **Out of scope**: no code changes to `analytics/sharpe.py` beyond the docstring (the three function bodies stay as `NotImplementedError("M4 deliverable")` stubs; M4 PR 1 lands those against the now-corrected contract). No scipy added to `pyproject.toml`. No changes to ADR 0001 (PSR/DSR/MinTRL non-optional decision stands).

### Fixed
- **SSGA XLSX real-structure parsing**: the first SSGA download revealed the actual export shapes differ from the initial loader assumptions. `spdr-product-data-us-en.xlsx` has a disclaimer paragraph in row 0, the group-header row (`As of**`, `Ticker`, ...) in row 1, the period sub-labels in row 2, and data from row 3; tickers carry a registered-trademark suffix (`SPY` reads as `SPY` + U+00AE, surfacing as `SPYM`/`SPY?`), which must be cleaned to avoid matching `SPYM`/`SPYG`; returns are percentage strings (`"31.01%"`) with `"-"` for not-applicable. `spdr-etf-historical-distributions.xlsx` stores ex-dates as `MM/DD/YYYY` text and dividends as whitespace-padded number-as-text. The loader now scans for the header row (handles the disclaimer offset), cleans tickers via leading-alnum regex, parses percent/number strings, and parses `MM/DD/YYYY`. Verified against Sam's real pull: 134 SPY dividends (1993-2026), SPY 10y NAV TR 15.1% as of 2026-04-30, correctly distinguished from the `SPYM`/`SPYG` rows. Synthetic test fixtures rewritten to reproduce all four quirks. New `SSGASpyReference.as_of_date` property exposes SSGA's trailing-return as-of date (needed for the window-alignment work in ADR 0006).
- **Adapter date dtype**: `SharadarDataSource.read_sep_prices` and `read_actions_dividends` now cast their date columns to `pl.Date` explicitly. The `nasdaq-data-link` SDK returns pandas `datetime64[ns]`; `pl.from_pandas` promotes that to `pl.Datetime`; `write_parquet` preserves it. Without the cast, downstream `(asset_id, dt)` price-index keys in the BarLoop become `(int, datetime)` while lookups use `dt.date()`, so every lookup silently returns `None`. The constant-weight demo then ran 3774 trading days, never rebalanced, stayed 100% in cash, and the reference function reproduced the same wrong number, so `engine_vs_reference: PASS` printed on `$0 vs $0`. This was a silent-correctness failure the synthetic-fixture tests did not catch because they wrote `pl.Date` columns directly. New regression test `test_read_sep_prices_returns_pl_date_column_even_from_datetime_source` writes a `pl.Datetime`-typed parquet and asserts the adapter returns `pl.Date`. Surfaced when Sam ran the demo against his first real Sharadar pull.
- **Manifest helper**: `sharadar_pull --refresh-hashes` now hashes `.parquet`, `.xlsx`, and `.csv` files (previously parquet-only). Required for the SSGA bundle which contains XLSX files. `row_count` is computed for parquets only; XLSX/CSV use `row_count=None` per the manifest schema. Updated `test_refresh_hashes_no_parquets_raises` to match the new error message.
